# 商家核销与推广系统 手工测试指南

这份指南用于在本地环境手工验证「商家核销与推广」全链路：**邀请追踪 → 激活发券 → 商家核销（动态二维码 + TOTP）→ 数据看板**。

系统涉及**三套相互独立的会话**，测试时建议用不同浏览器 / 无痕窗口分别保持登录，避免串号：

- 普通用户（`/dashboard/*`，`lilink_token` cookie）
- 后台运营（`/admin/*`，`lilink_admin_token` cookie）
- 商家核销（`/merchant/*`，`lilink_merchant_token` cookie）

与破冰测试不同，本系统的端到端测试数据需要按下面的「数据准备」手动配置一次；如只需预置优惠券 demo 数据，可使用 `npm run db:seed-coupon-demo`。核销重构见 `docs/merchant-redemption-qr-totp-redesign.zh-CN.md`，优惠券阶梯规则见 `docs/merchant-coupon-rule-and-redemption.zh-CN.md`。

## 前置条件

1. 确认 `apps/api/.env` 已配置，**必须包含**两个核销相关 secret（各 ≥16 字符，互不相同，也与 `JWT_SECRET`/`ADMIN_JWT_SECRET` 不同）：
   - `MERCHANT_JWT_SECRET`：商家会话。
   - `REDEEM_TICKET_SECRET`：核销「确认票据」（prepare 阶段签发的 3 分钟一次性 JWT）。
   缺任一项时 API 启动会在环境解析阶段直接失败。
2. 确认 PostgreSQL 正在运行：

```sh
npm run infra:up
```

3. 应用 migration（含 merchant 系统的表、`Coupon.totpSecret`、`Redemption.giftLabel`）：

```sh
npm run db:migrate:deploy
```

4. 灌入默认数据（学校域名、问卷、匹配周期）：

```sh
npm run db:seed-defaults
```

5. 创建后台管理员（首次）：

```sh
cd apps/api && node scripts/bootstrap-admin.mjs
```

6. 启动本地服务：

```sh
npm run dev
```

默认访问地址：

- Web：`http://localhost:3000`
- API：`http://localhost:4000/v1`
- 后台：`http://localhost:3000/admin`
- 商家核销：`http://localhost:3000/merchant/login`

## 角色与账号

| 角色 | 登录入口 | 来源 |
|------|----------|------|
| 运营管理员 | `/admin` | `bootstrap-admin.mjs`（看 `apps/api/.env` 的 `ADMIN_BOOTSTRAP_*`） |
| 邀请人「A」 | `/login` | 自行注册的普通用户 |
| 被邀请人「B」 | `/register` | 通过 A 的邀请链接注册 |
| 商家账号 | `/merchant/login` | 在场景 A 由管理员在后台创建 |

建议邮箱用支持的学校域名（见 `seed-defaults` 灌入的学校），否则注册会被拒。

测试核销时，准备**两台设备 / 两个窗口**：一个是 **B 的用户端出码页**（手机或一个浏览器窗口），一个是**商家端**（另一浏览器，已登录 `/merchant`）。核销成功后两端展示不同内容，需同时观察。

## 场景 A：后台配置（商家 + 账号 + 活动 + 券包）

> 这是其余场景的数据基础，先做一次。

1. 打开 `/admin`，用管理员登录。
2. **建商家**：进入「商家」（`/admin/merchants`），输入商家名称 + 联系方式，点「新增商家」。
3. **建商家账号**：点该商家的「账号 / 推广位」展开，填邮箱 + 初始密码（≥8 位）+ 角色（店主/店员），点「新增账号」。记下这个邮箱和密码（场景 D 用）。
4. **填推广位**（**用户端**核销成功后展示，引导关注商家）：在同一展开区的「核销成功页推广位（JSON）」里填，例如：

```json
[
  { "type": "TEXT", "text": "关注公众号领更多福利" },
  { "type": "QRCODE", "imageUrl": "https://example.com/qr.png", "caption": "扫码关注" }
]
```

点「保存推广位」。图片 URL 必须是 https。注意：这些推广位现在展示在**用户核销成功后的用户端页面**，不再显示在商家端（见场景 D）。

5. **建活动**：进入「活动券包」（`/admin/campaigns`），填活动名称 + slug（小写字母数字短横，如 `spring-2026`），点「新建活动」。
6. **建券包**：点该活动的「券包」展开，选刚建的商家、填券标题（如 `满50减10`）、选优惠类型、填**名义面值（元）**、可选有效天数，点「新增券模板」。
   - 想测满减档位 / 消费金额求值，建一个**带门槛的分档券**（如满 50 减 10）；想测无门槛直减，建一个固定面值券。
7. **上线活动**：在活动卡片的状态控件选 `ACTIVE`、勾「设为默认」，点「应用状态」。

**验证**：

- 活动显示 `ACTIVE` 且「默认」。
- 券包里能看到刚建的券模板（面值按元正确显示）。
- 商家账号、推广位已保存。

> 关键：活动必须在拉新前置为 `ACTIVE` 且默认，否则更早注册的用户没有归属、激活也不会发券。

## 场景 B：邀请追踪（个人码 → 落地页 → 注册冻结归属）

1. 注册并登录用户 **A**，进入 `/dashboard/me` →「我的邀请」（`/dashboard/referrals`）。
2. 验证：能看到 A 的个人邀请码、二维码、各渠道分享按钮、邀请漏斗（初始全 0）。
3. 点任一渠道「复制链接」，验证出现「分享链接已复制」toast，链接形如 `http://localhost:3000/i/<10位码>?ch=<渠道>`。
4. **无痕窗口**打开该链接：验证落地页短暂停留后跳转到 `/register`。
5. 在该无痕窗口注册用户 **B**（用另一个学校邮箱）。
6. 回到 A 的 `/dashboard/referrals`，刷新。

**验证**：

- A 的漏斗「已邀请注册」+1。
- （可后台核对）B 的来源活动被冻结为场景 A 的活动。渠道两层：**来源（source）= 个人邀请（PERSONAL，因为有 `referredByUserId`）**；**分享信息**由链接里的 `ch` 决定，后台/看板会拆成媒介（medium：微信/链接/二维码/其他）+ 场景（scene：朋友圈/群/私聊，仅微信有）。`?ch=` 参数保持原样、向后兼容。

## 场景 C：激活发券（完善资料 + 报名 → 自动发可用券）

1. 用 **B** 登录，进入 `/dashboard/profile`，完整提交问卷（完善资料）。
2. 进入 `/dashboard/intent`，**报名当前匹配周期**（首次 opt-in）。若没有可报名的周期，先到 `/admin/cycles` 确认有 `OPEN` 周期（必要时新建/开启）。
3. 进入 `/dashboard/coupons`。

**验证**：

- B 的「可用」分区出现券：含商家名、优惠文案、有效期，以及「**出示核销码**」入口（不再是静态明文码，见场景 D）。
- A 的 `/dashboard/referrals` 漏斗「已激活」「已领券」+1。

> 激活 = 问卷 `submittedAt` + 首次 opt-in（`firstOptedInAt`）同时满足；满足后系统**自动发可用券（ISSUED）**，没有手动领取步骤。只完善资料或只报名都不会发券。发券时每张券会生成独立的 TOTP seed（`totpSecret`），供出码用。

## 场景 D：商家核销（动态二维码 + TOTP，两步确认）

新核销流程：**用户出示动态二维码 / 文本码 → 商家扫码或手输 → 确认页（可输消费金额）→ 确认核销**。核销成功后**商家端只看到成功 + 金额**，**用户端**自动切换显示商家推广位。

### D1. 用户出码

1. 用 **B**（用户端）打开 `/dashboard/coupons`，在可用券上点「**出示核销码**」。
2. **验证**：弹出 ① 动态**二维码** ② 文本核销码 `<6位定位码>-<6位数字>`（如 `K7M2QP-573821`）③ 刷新倒计时。等约 60s，验证二维码与文本码**同步刷新**（动态码，防截图转卖）。

### D2. 商家扫码核销（主路径）

3. 商家端已登录 `/merchant`（场景 A 账号）。用任意扫码工具扫 B 的二维码，得到形如 `http://localhost:3000/r/<code>#t=<totp>` 的网址，在**已登录商家账户的浏览器**打开。
4. **验证确认页**：显示券标题、优惠、面值、持券人。**满减档位券**会出现「消费金额」输入框——填一个达到门槛的金额（如满 50 减 10，填 ≥50 元）。
5. 点「**确认核销**」。
   - **商家端验证（SUCCESS）**：显示「✓ 核销成功」+ 实际优惠 / 消费金额 / 赠品（若有）。**不再显示商家推广位**。
   - **用户端验证**：B 的出码页约 2–3 秒内（轮询）**自动切换**为「核销成功」，显示优惠结果 + **商家电话 / 公众号二维码**（场景 A 配置的推广位）。

### D3. 手动输码后备

6. 让 B 出码页读出**当前**文本码 `<code>-<totp>`；商家在 `/merchant/redeem` 输入框输入它。
7. **验证**：与扫码等价 → 进入同一确认页 → 确认核销成功。（手动码同样是动态的，过期同样失效。）

### D4. 防转卖：TOTP 过期

8. 截图 B 某张券的二维码，等 **>2 分钟**后再扫该截图（或用过期的 `#t=`）。
9. **验证（EXPIRED_CODE）**：提示「二维码已过期，请让用户刷新后重试」。说明动态码在约 1–2 分钟（含 ±1 时间窗容错）后失效，旧截图无法核销。

### D5. 重复核销

10. 对**已核销**的券再次出码并核销（扫码或手输）。
11. **验证（ALREADY_USED）**：提示已使用，不进入核销。一张券只能成功核销一次（即使确认票据在 3 分钟有效期内重复提交，也由 CAS 兜底，只核销一次）。

### D6. 满减门槛

12. 对满减档位券，在确认页填一个**低于门槛**的金额（如满 50 减 10，填 30 元）。
13. **验证（BELOW_THRESHOLD）**：提示未达门槛、不核销。不填金额则提示需要输入金额（NEED_AMOUNT）。

### D7. INVALID（不泄露）

14. 分别测试：① 用 A 商家的券去 B 商家核销（跨商家）；② 管理员在 `/admin/users` 把 B 置为 SUSPENDED 后核销其券；③ 手输乱码 / 格式不对的码。
15. **验证（INVALID）**：一律 INVALID，不泄露券是否存在 / 归属。格式不对（非 `定位码-6位数字`）的手输码本地直接拦截。

### D8. 未登录扫码回跳

16. 商家端**退出登录**，再扫 B 的二维码打开 `/r/<code>`。
17. **验证**：跳转 `/merchant/login?next=/r/<code>`；登录后回到 `/r/<code>`，因 totp 在 hash fragment 中、跳转后已丢失，页面提示「请让用户重新出示二维码并再次扫码」。

**本场景验证小结**：

- 双端分工：商家端只见成功 + 对账金额；用户端见优惠结果 + 商家推广位。
- 动态码 ~1–2 分钟失效；一券一次；跨商家 / 非 ACTIVE / 不存在 一律 INVALID 不泄露。
- 满减券走「确认页输金额」，无门槛券直接确认。

## 场景 E：数据看板

1. 打开 `/admin/promotion`，选择场景 A 的活动 + 覆盖测试时间的日期范围，点「查询」。
2. **拉新漏斗**：验证分享 / 点击 / 注册 / 激活 / 领券 / 核销六步计数，以及分性别表格与相邻转化率。
3. **渠道分解（channelBreakdown）**：验证漏斗下方出现按**媒介 → 场景**分组的渠道分布表（微信→朋友圈/群/私聊，链接，二维码，其他），数字与场景 B 用过的 `ch` 一致。注意表下「仅展示已标记渠道，合计可能小于分享/点击总数」的说明——无 `?ch=` 的旧链接/直访点击不计入分解。
4. **邀请排行榜**：来源筛选有**两个**——「个人码（PERSONAL）/ 默认活动（DEFAULT）」。验证 A 出现在个人码榜；无个人来源、仅默认活动归属的用户出现在「默认活动」榜（展示为「默认活动」而非 `DEFAULT`）。邀请 / 激活 / 领券 / 核销与性别分桶正确；超过 20 条翻页可用。
5. **券情况**：按商家显示发放 / 核销数。
6. **核销对账**：按商家 + 天（Asia/Shanghai）显示核销单数与面值合计；翻页可用。

**验证**：

- 注册数 = 该活动归属、非测试用户；漏斗「核销」与排行榜 `redeemed` 是**去重人数**，而券情况 / 对账是**核销单数**，两套口径不混。
- 把日期范围调到 00:00–08:00（中国时间）边界，验证当天数据按上海日归属，不漏不重。
- API 会把 source 参数大小写归一化，但语义上只接受 PERSONAL/DEFAULT；前端发送这两个大写值。

## 场景 F：停用边界

1. **停用商家**：在 `/admin/merchants` 停用场景 A 的商家，验证该商家账号登录 / 核销被拒。
2. **停用账号**：重新启用商家后，单独停用某个商家账号，验证该账号登录被拒，但同商家其它启用账号正常。

## 检查重点

- 三套会话严格隔离：用户、后台、商家各用独立 cookie 与 secret；互相不能越权访问。
- 归属在**注册时冻结**到 `referralCampaignId`，不随后续活动变更漂移。渠道分两层：来源（PERSONAL/DEFAULT，由 `referredByUserId` 是否存在派生）+ 分享信息（媒介/场景，由 `channel` 派生）。
- 激活 = 问卷提交 + 首次 opt-in；自动发 `ISSUED` 券（含 `totpSecret`），无手动领取。
- 核销是**两步**：`prepare`（校验券 + 验当前 TOTP 新鲜度，签 3 分钟一次性票据，不改状态）→ `redeem`（验票据 + CAS 置 `REDEEMED`）。动态码约 1–2 分钟失效以防截图转卖。
- 只有 `status==ACTIVE` 用户的券可核销；一券一次（CAS 兜底重放）；INVALID 不泄露券是否存在。
- **满减档位券**会在确认页要求输入消费金额并求值（NEED_AMOUNT / BELOW_THRESHOLD）；无门槛券直接确认。`Redemption` 记录消费金额、实际优惠、赠品快照（`giftLabel`）。
- 商家端核销成功**不展示**推广位；推广位只在**用户端**核销成功视图展示（电话 / 公众号二维码）。
- `prepare`/`redeem` 有路由级节流（防暴力猜码）；TOTP 进二维码的 hash fragment，不入服务器日志。
- `isTest` 用户全程从看板统计中排除。
- 看板均按 `campaignId` + 时间范围（Asia/Shanghai 业务日）过滤。

## 常见问题

- API 启动报缺 `MERCHANT_JWT_SECRET` 或 `REDEEM_TICKET_SECRET`：在 `apps/api/.env` 补齐这两个变量（各 ≥16 字符）后重启。
- `/dashboard/coupons` 没有券：确认 B 已「完善资料 + 报名匹配周期」，且其来源活动是场景 A 的 `ACTIVE` 默认活动（注册早于活动上线则没有归属、不发券）。
- 出码页二维码不显示：核销出码页用**内置 `qrcode` 库**渲染（不依赖外网）；若空白多为前端报错，看浏览器控制台。（`/dashboard/referrals` 的分享二维码仍可能用外部服务，需网络可达。）
- 扫码打开提示「二维码已过期」：动态码 ~1–2 分钟失效，让用户在出码页刷新后重新出示 / 重新扫；或商家端未登录被回跳后 totp 丢失，按提示让用户重扫。
- 满减券核销失败：确认在确认页填了**达到门槛**的消费金额；不填或低于门槛会被拒。
- 商家登录失败：确认商家与该账号都处于启用状态；停用任一方都会拒绝登录与核销。
- 看板为空：确认选了正确活动、时间范围覆盖数据；测试账号（`isTest`）会被排除。
- 渠道分解合计对不上分享/点击总数：正常，未标记 `?ch=` 的事件不计入分解（见场景 E 说明）。
- 切换角色时串号：用无痕窗口，或先退出当前会话再登录另一角色。
