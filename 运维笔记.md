## 一、本地如何测试

### 前置条件

- Node.js（版本以仓库 `engines` 为准）、Docker Desktop。
- 将 `apps/api/.env.example` 复制为 `apps/api/.env`（可选：仓库根目录再建 `.env`；`scripts/load-env.mjs` 先读根目录，再由 `apps/api/.env` 覆盖）。
- `DATABASE_URL` 指向本地 Postgres（与 `docker-compose.yml` 一致，示例：`postgresql://lilink:lilink@127.0.0.1:5432/lilink?schema=public`）。
- `JWT_SECRET`、`ADMIN_JWT_SECRET`、`CRON_SECRET` 须**互不相同**，且各**至少 16 字符**。
- `ADMIN_BOOTSTRAP_EMAIL` / `ADMIN_BOOTSTRAP_PASSWORD`（密码**至少 12 字符**）用于 `/admin`；本地邮件调试：`SMTP_HOST=127.0.0.1`、`SMTP_PORT=1025`，Mailpit 需 compose profile **`dev-mail`**（`npm run infra:up` 已包含）。

### 初始化与启动（仓库根目录）

```bash
npm install
npm run infra:up
npm run db:generate
npm run db:bootstrap
npm run db:seed          # 全量种子（含演示用户）；仅需结构+OPEN 轮次时用 db:seed-defaults
cd apps/api && node scripts/bootstrap-admin.mjs && cd ../..
npm run dev
```

### 访问地址

| 服务 | 地址 |
|------|------|
| Web | http://localhost:3000 |
| API 健康检查 | http://localhost:4000/v1/health |
| OpenAPI（非生产） | http://localhost:4000/docs |
| Mailpit | http://localhost:8025 |

### 业务验收（匹配一轮）

与第二节流程相同，区别是本地 API 跑在宿主机，**不要**使用 `docker exec lilink-api`。

1. 普通用户：测试账号与密码见**第二节**；管理员用 `ADMIN_BOOTSTRAP_*` 登录 `/admin`。
2. `/admin` → **轮次** → 选状态为 **OPEN** 的轮次 → **强制执行**。
3. 用 Alice / Bob / Carol（或等价种子账号）登录 `/dashboard` 核对匹配结果。

### API 自动化测试

Postgres 已启动且迁移已应用：`npm run test --workspace api`。

## 三、日常维护问题

### 部署与发布（生产机）

**前置**：代码在约定目录；`.env` 或 `apps/api/.env` 已配置且**勿提交 Git**；Docker 可用；反向代理（如 Caddy）指向 `127.0.0.1:4000`，改配置后需 `sudo systemctl reload caddy`。

**推荐顺序**（在 `/home/admin/lilink`）：

1. **回滚准备**：构建新镜像前为当前运行中版本打标签（见下节「API 镜像回滚」）。
2. `git pull`（或既定发布流程）。
3. `docker compose --profile prod build api` → `docker compose --profile prod up -d`。
4. 若含 Prisma 迁移：`docker exec lilink-api npx prisma migrate deploy`。
5. `docker logs --tail 100 lilink-api` 与 `curl -I https://api.lilink.top/v1/health` 验证。

本地仅起数据库：`docker compose up -d`；需 Mailpit：`npm run infra:up` 或 `docker compose --profile dev-mail up -d`。

### 仅重启 API（不重建镜像）

```bash
cd /home/admin/lilink
docker compose --profile prod up -d --force-recreate api
```

### API 镜像回滚

线上跑的是 **Docker 镜像**，与 Git 分支无关；要立刻回到上一版后端，应恢复**上一版镜像**。若坏版本已执行**不可逆**迁移，仅换镜像可能不够，需迁移方案或从「数据库备份」恢复。

**上线前**（每次 `build api` 之前）保存当前线上镜像（镜像名 `lilink-api:latest`）：

```bash
cd /home/admin/lilink
docker inspect -f '{{.Image}}' lilink-api | xargs -I{} docker tag {} lilink-api:previous-good
```

无容器时可用：`docker tag lilink-api:latest lilink-api:previous-good`；无任何镜像则跳过。

**回滚**（勿先执行 `build`）：

```bash
cd /home/admin/lilink
docker tag lilink-api:previous-good lilink-api:latest
docker compose --profile prod up -d --force-recreate api
```

验证：`docker logs --tail 100 lilink-api` 与健康检查 URL。

**长期**：重要发布可打 Git tag；使用镜像仓库时以版本号标签（如 `registry/lilink-api:20260412-1`）发布与回滚更稳。

### 数据库

```bash
# 迁移（容器内）
docker exec lilink-api npx prisma migrate deploy

# 默认种子（学校 + 问卷 + OPEN 轮次，无用户）
docker exec lilink-api node scripts/seed-defaults.mjs

# 全量种子（含测试用户，仅开发/演练；服务器上通常需本地有依赖与源码）
# npm run db:seed

# 手动备份 / 恢复
bash scripts/backup-postgres.sh
gunzip -c /home/admin/backups/lilink/<file>.sql.gz \
  | docker exec -i lilink-postgres psql -U lilink -d lilink
```

定时备份：北京时间每日 3:00，保留 7 天，目录 `/home/admin/backups/lilink/`。

路径：`/admin/users` — **标记为测试**、筛选（全部 / 真实 / 测试）、**删除全部测试用户**。

### 群发提醒邮件（已填问卷但未参与）

对已提交问卷但未在目标 OPEN 轮次中 opt-in 的用户发送提醒邮件。默认选最早创建的 OPEN 轮次。

```bash
cd /home/admin/lilink/apps/api

# 预览（不发送）
DATABASE_URL="postgresql://lilink:lilink@127.0.0.1:5432/lilink?schema=public" \
  node scripts/remind-non-participants.mjs --dry-run

# 正式发送
DATABASE_URL="postgresql://lilink:lilink@127.0.0.1:5432/lilink?schema=public" \
  node scripts/remind-non-participants.mjs

# 指定轮次
DATABASE_URL="postgresql://lilink:lilink@127.0.0.1:5432/lilink?schema=public" \
  node scripts/remind-non-participants.mjs --codename=第一周
```

同一轮次对同一用户有 `dedupeKey` 去重，重复执行不会重复发送。
