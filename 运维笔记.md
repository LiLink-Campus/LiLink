## 一、应急剑谱（出事后照着操作）

> 每个场景固定模板：**症状 → 立即缓解 → 排查 → 根因 → 验收**。
> 任何修复操作前先做一次 `bash scripts/backup-postgres.sh`，3 秒成本买一份保险。

### 1.1 API 挂了 / 502 / 健康检查失败

**症状**

- 浏览器访问 `https://api.lilink.top/...` 拿到 502 / 超时。
- `curl https://api.lilink.top/v1/health` 非 200。
- 用户大面积反馈"打不开"。

**立即缓解（30 秒内尝试）**

```bash
cd /home/admin/lilink
# 先看容器是否还在跑
docker ps --format "table {{.Names}}\t{{.Status}}"
# 容器没挂就 force-recreate；容器挂了同一条命令也能重建
docker compose --profile prod up -d --force-recreate api
sleep 5
curl -sS -o /dev/null -w "HTTP %{http_code}\n" http://127.0.0.1:4000/v1/health
```

如果 `force-recreate` 仍不健康，**直接进 1.6 镜像回滚**。

**排查**

```bash
# 1. API 容器最近日志（看是不是启动报错或 panic）
docker logs --tail 200 lilink-api 2>&1 | tail -80
# 2. 是不是数据库连不上
docker ps lilink-postgres
docker exec lilink-postgres pg_isready -U lilink
# 3. 反代是否在跑
systemctl is-active caddy
# 4. 容器内 vs 容器外都试一遍，定位"是 API 还是反代"
curl -sS -o /dev/null -w "container: %{http_code}\n" http://127.0.0.1:4000/v1/health
curl -sIk https://api.lilink.top/v1/health | head -1
```

**根因 → 修复**

| 根因 | 修复 |
|------|------|
| 新版本镜像有 bug | **1.6 回滚到 `previous-good`** |
| Postgres 不健康 | 看 1.2 |
| Caddy 配置改坏 | `sudo systemctl reload caddy`；坏了从 git 或 `Caddyfile.bak` 恢复后再 reload |
| 容器 OOM / 资源不足 | `docker stats`、`free -h`、`df -h /mnt/data`；按需扩容或清理 |
| 数据盘没挂 → dockerd 拉不起容器 | 看 1.5 |

**验收**

```bash
curl -sS http://127.0.0.1:4000/v1/health        # 容器内 200
curl -sIk https://api.lilink.top/v1/health      # 外部 200
docker logs --tail 50 lilink-api                # 无 error
```

### 1.4 磁盘满了

**症状**

- `docker compose build` 报 `no space left`。
- API 写日志失败、Postgres 写入卡住。

**立即定位 — 是哪块盘？**

```bash
df -h / /mnt/data
```

**根因 → 修复**

| 满的盘 | 大概率根因 | 修复 |
|--------|------------|------|
| `/`（系统盘 vda3） | `/var/log` 暴涨；`/var/lib/docker.old` `.containerd.old` 没清；用户 home 大文件 | `sudo journalctl --vacuum-time=7d`；`sudo rm -rf /var/lib/docker.old /var/lib/containerd.old`；`du -sh /var/log/* \| sort -h \| tail` |
| `/mnt/data`（数据盘 vdb） | Docker 镜像 / 构建缓存堆积；备份历史太多 | 见下"清 Docker"；`ls -lt /mnt/data/backups/postgres \| tail` 删旧备份 |

**清 Docker（保留 `previous-good` 不要清）**

```bash
docker images lilink-api                 # 确认 previous-good 还在
docker builder prune -af                 # 构建缓存（最大头）
docker image prune -f                    # 悬空镜像（不动有 tag 的）
df -h /mnt/data
```

**验收**

```bash
df -h / /mnt/data
docker compose --profile prod up -d --build api    # 能正常 build 即为通
```

### 1.5 数据盘掉了 / 重启后未挂载

**症状**

- 服务器重启后 `docker ps` 报 `Cannot connect to the Docker daemon`。
- `df -h` 看不到 `/mnt/data`。
- API 容器起不来。

**立即排查**

```bash
lsblk                              # /dev/vdb 是否还在
cat /etc/fstab | grep vdb          # fstab 里 UUID 是否还对
sudo blkid /dev/vdb                # 当前实际 UUID
sudo mount -a                      # 按 fstab 重新挂一次
```

**根因 → 修复**

| 根因 | 修复 |
|------|------|
| `mount -a` 后已挂上 | 完事；如反复发生检查 `fstab` 是否漏 `nofail` |
| UUID 不一致（云厂商替换了盘） | 用 `blkid` 取新 UUID 改 `/etc/fstab`，再 `sudo systemctl daemon-reload && sudo mount -a` |
| 盘真的丢了 / 故障 | 走云厂商工单；从最近 Postgres 备份在新盘上恢复（见 1.2） |

**挂上后必做**

```bash
df -h /mnt/data                              # 应 /dev/vdb 在
docker info | grep "Docker Root Dir"         # 应 /mnt/data/docker
sudo systemctl restart containerd docker     # 必要时重启
docker compose --profile prod up -d
```

## 二、生产运维（计划内）

### 2.1 服务拓扑与磁盘布局

生产机使用两块磁盘，**Docker 与备份均落在数据盘**，避免系统盘膨胀：

| 设备 | 挂载点 | 用途 | 关键路径 |
|------|--------|------|----------|
| `/dev/vda3` | `/` | 系统、代码、日志 | `/home/admin/lilink/`（代码仓） |
| `/dev/vdb` | `/mnt/data` | Docker + 备份 | `/mnt/data/docker`、`/mnt/data/containerd`、`/mnt/data/backups` |

**关键配置**

- `/etc/docker/daemon.json` 设 `"data-root": "/mnt/data/docker"`。
- `/etc/containerd/config.toml` 设 `root = "/mnt/data/containerd"`（dockerd 用外部 containerd，镜像层数据落在这里）。
- `/etc/fstab` 用 UUID 持久化挂载 vdb，`nofail` 保证数据盘故障不阻塞开机。
- `/home/admin/lilink/backups` 是软链 → `/mnt/data/backups`，保持现有脚本路径不变。

**容量监控**：`df -h /` 与 `df -h /mnt/data`；构建多次后空间紧张走 [1.4 磁盘满了](#14-磁盘满了)。

**迁移残留（如仍存在可手动清理）**

```bash
# 确认业务稳定后再删（建议至少跑过一个完整业务周期 / 一次匹配轮次）
sudo rm -rf /var/lib/docker.old /var/lib/containerd.old
```

**重启数据盘自动挂载校验**

```bash
df -h /mnt/data                         # 应见 /dev/vdb 挂在 /mnt/data
docker info | grep "Docker Root Dir"    # 应见 /mnt/data/docker
docker ps                                # lilink-api / lilink-postgres 都应 Up
```

### 2.4 镜像回滚

线上跑的是 **Docker 镜像**，与 Git 分支无关；要立刻回到上一版后端，应恢复**上一版镜像**。若坏版本已执行**不可逆**迁移，仅换镜像可能不够，需迁移方案或从「数据库备份」恢复。

**上线前**（每次 `build api` 之前）保存当前线上镜像（镜像名 `lilink-api:latest`）：

```bash
cd /home/admin/lilink
docker inspect -f '{{.Image}}' lilink-api | xargs -I{} docker tag {} lilink-api:previous-good
```

无容器时可用：`docker tag lilink-api:latest lilink-api:previous-good`；无任何镜像则跳过。

**回滚**（勿先执行 `build`）：

```bash
cd /home/admin/lilink
docker tag lilink-api:previous-good lilink-api:latest
docker compose --profile prod up -d --force-recreate api
```

**验证**：`docker logs --tail 100 lilink-api` 与健康检查 URL。

**长期建议**：重要发布可打 Git tag；使用镜像仓库时以版本号标签（如 `registry/lilink-api:20260412-1`）发布与回滚更稳。

### 2.6 邮件服务（SMTP）

**当前生效**：阿里云邮件推送（DirectMail），SMTP 通道；按量 2 元/千封，5k 用户全年成本约 100 元。

关键 `.env`：

```bash
SMTP_HOST=smtpdm.aliyun.com
SMTP_PORT=465
SMTP_SECURE=true
SMTP_USER=noreply@lilink.top          # 必须与 SMTP_FROM 邮箱地址一致（阿里云强制）
SMTP_PASS=<阿里云控制台 → 发信地址 → 设置 SMTP 密码>
SMTP_FROM=LiLink <noreply@lilink.top>
SMTP_SEND_CONCURRENCY=20              # 默认 10；阿里云共享 IP 单账号 ~30 QPS 内安全
SMTP_MAX_CONNECTIONS=20               # 建议与并发一致
```

**切换 SMTP 服务商的标准流程**

1. 改 `apps/api/.env`（根目录 `.env` 是软链）。
2. **预检**（不重启 API，用一次性脚本验证鉴权与入队，失败不会影响在跑容器）：

   ```bash
   docker exec -i \
     -e SMTP_HOST=smtpdm.aliyun.com -e SMTP_PORT=465 -e SMTP_SECURE=true \
     -e SMTP_USER=noreply@lilink.top -e SMTP_PASS='<smtp-password>' \
     -e SMTP_FROM='LiLink <noreply@lilink.top>' \
     -e TEST_TO=<your-mailbox> \
     lilink-api node <<'EOF'
   const nodemailer = require('nodemailer');
   (async () => {
     const port = Number(process.env.SMTP_PORT);
     const t = nodemailer.createTransport({
       host: process.env.SMTP_HOST, port,
       secure: process.env.SMTP_SECURE === 'true' || port === 465,
       auth: { user: process.env.SMTP_USER, pass: process.env.SMTP_PASS },
       connectionTimeout: 10000, greetingTimeout: 10000, socketTimeout: 15000,
     });
     await t.verify();
     const info = await t.sendMail({
       from: process.env.SMTP_FROM, to: process.env.TEST_TO,
       subject: 'LiLink SMTP doctor', text: 'OK if you can read this.',
     });
     console.log('OK', info.messageId, info.response);
     t.close();
   })().catch((e) => { console.error('FAIL', e.message, e.code, e.response); process.exit(1); });
   EOF
   ```

3. 预检 PASS 后重启容器：`docker compose --profile prod up -d --force-recreate api`。
4. 业务验证：用未注册的合规学校邮箱调一次 `POST /v1/auth/request-code`，期望 HTTP 201 且**响应 <200 ms**（接口已改为 fire-and-forget，不阻塞 SMTP）；稍等 1~2 秒后查 DB，`OutboundEmail.status` 从 `PENDING` 翻到 `SENT` 即通。

**投递行为**

cron 每 30 秒扫一次（`mail.service.ts#handleEmailQueue`）；业务侧 `auth.service` 对验证码邮件走 **fire-and-forget**（`void deliverQueuedEmailNow(...)`），HTTP 秒回，真实投递失败由 cron 按 `maxAttempts` 重试兜底。因此 **HTTP 200 ≠ 收件人已收到**，真要确认投递需查 `OutboundEmail.status` 或阿里云控制台「邮件追踪」。

`OutboundEmail.status` 含义见 [附录 C](#c-outboundemail-状态字段)。

**避坑**

- **新发信域名**（如 `lilink.top`）首次发到陌生邮箱常落 Spam；让首批用户手动"非垃圾邮件"训练一段时间。
- 不要用 163 / QQ 个人邮箱做生产 SMTP：日上限 700~1400 封、分钟级速率约 10~30 封，集中注册期会大量 503。
- DNS 必须配置 SPF + DKIM；阿里云控制台「发信域名」会标红未通过的项。

### C. OutboundEmail 状态字段

| status | 含义 |
|--------|------|
| `PENDING` | 入队待发 |
| `PROCESSING` | 正在发送（>10 min 未结束会被视为"过期"由 cron 重新抢占） |
| `SENT` | SMTP 已 250 入队上游 |
| `FAILED` | 投递失败，等 cron 重试（退避 `attemptNumber × 60s`） |
| `EXHAUSTED` | 已达 `maxAttempts`（验证码 3 次、引荐邮件 5 次） |
